# Zod v3 → v4 Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate `notion-mcp-server` from `zod@^3.24.2` to `zod@^4.4.3` with no behavior changes visible to MCP clients, shipping as v1.4.0.

**Architecture:** Zod 4 is a non-breaking move for *most* of this codebase — every schema continues to compile, every `z.infer<...>` continues to resolve, every `safeParse()` continues to work. The breaking changes that *do* affect us are surgical: deprecated `.email()`/`.url()` method-form string formats (5 call sites), and one `.refine({ message })` key rename to `.refine({ error })` (1 call site). MCP SDK `1.29.0` already advertises `zod@^3.25 || ^4.0` in its peer-deps, so no SDK bump is required. The Notion client (`@notionhq/client@2.3.0`) does not depend on Zod at all.

**Decision: standard `zod` v4, NOT `zod/mini`.**

| Factor | `zod` v4 | `zod/mini` |
|---|---|---|
| API style | Method chains (`schema.optional().array()`) — what the codebase uses today | Functional wrappers (`z.array(z.optional(schema))`) — requires rewriting every schema file |
| Bundle size | ~5.4 KB gzipped core | ~1.9 KB gzipped core |
| Server-side relevance | Bundle size does not matter — we ship a ~183 MB Node Docker image and run `node build/index.js` directly | n/a |
| MCP SDK compatibility | Officially supported in 1.29.0 | Not advertised; would need MCP SDK call-site validation |
| Migration effort | ~6 line changes total | Full rewrite of all 20 schema files |

The functional API of `zod/mini` exists to enable browser/edge tree-shaking. We run inside Node on a server. Choosing `zod/mini` would mean rewriting every `.optional()`/`.array()`/`.default()` call across `src/schema/` and `src/types/` for zero benefit to anyone who actually runs this server.

**Tech Stack:** Zod 4.4.3, TypeScript 5.8.2, MCP SDK 1.29.0, Node 22 (Docker) / Node ≥20 (engines).

**Out of scope:** Adopting top-level format APIs beyond what's needed (e.g., `z.iso.datetime()`); adopting `z.strictObject()` / `z.looseObject()` (the codebase doesn't call `.strict()`/`.passthrough()` anywhere); adopting the new error customization API for new schemas (only the one existing `message:` site gets touched); adding a test framework (the project has no test infrastructure — validation runs through `tsc`, Docker rebuild, and an MCP `initialize` smoke test we established when shipping v1.3.0).

## Task 4: Migrate `src/schema/icon.ts` refine — rename `message` → `error`

The only Zod-4-incompatible call in `src/schema/icon.ts` is `{ message: "Invalid emoji" }` as the second argument to `.refine()`. Zod 4 renamed this key to `error`.

Verified by reading the file: the refine predicate is `(value) => /(\p{Emoji}️|\p{Emoji_Presentation})/gu.test(value)` — no `value is X` type guard, so Zod 4's tightened refine-narrowing semantics do not affect this file. The downstream `.transform((value) => value as EmojiRequest)` performs the type assertion explicitly.

**Files:**
- Modify: `src/schema/icon.ts:10`

Actual current file content (for reference — do not paste; only the `message:` key changes):

```ts
import { EmojiRequest } from "../types/emoji.js";
import { z } from "zod";

export const ICON_SCHEMA = z.object({
  emoji: z
    .string()
    .refine(
      (value) => /(\p{Emoji}️|\p{Emoji_Presentation})/gu.test(value),
      {
        message: "Invalid emoji",
      }
    )
    .transform((value) => value as EmojiRequest),
  type: z.literal("emoji"),
});
```

- [ ] **Step 1: Rename `message:` → `error:`**

In `src/schema/icon.ts`, change line 10 from:

```ts
        message: "Invalid emoji",
```

to:

```ts
        error: "Invalid emoji",
```

(Preserve the 8 leading spaces — the key sits inside `{ ... }` which sits inside `.refine(predicate, { ... })`.)

- [ ] **Step 2: Build and confirm green**

Run:

```bash
npm run build
```

Expected: no TS errors.

- [ ] **Step 3: Runtime smoke-check `ICON_SCHEMA` accepts a valid emoji and rejects an invalid one with the renamed error**

The project is `"type": "module"` in `package.json`, so use dynamic `import()`, not `require()`:

```bash
node --input-type=module -e "
import('./build/schema/icon.js').then(async ({ ICON_SCHEMA }) => {
  const ok = ICON_SCHEMA.safeParse({ emoji: '😀', type: 'emoji' });
  console.log('valid:', ok.success, '→', ok.success ? ok.data : ok.error.issues[0]?.message);
  const bad = ICON_SCHEMA.safeParse({ emoji: 'not-an-emoji', type: 'emoji' });
  console.log('invalid:', bad.success, '→', bad.success ? bad.data : bad.error.issues[0]?.message);
  if (ok.success !== true) process.exit(1);
  if (bad.success !== false) process.exit(1);
  if (bad.error.issues[0]?.message !== 'Invalid emoji') {
    console.error('FAIL: expected error message \"Invalid emoji\", got', bad.error.issues[0]?.message);
    process.exit(1);
  }
  console.log('OK');
});
"
```

Expected output:

```
valid: true → { emoji: '😀', type: 'emoji' }
invalid: false → Invalid emoji
OK
```

If the invalid-case error message is *anything other than* `Invalid emoji`, the `error:` rename did not land — re-check step 1.

- [ ] **Step 3: Send a `tools/list` request to confirm tool schemas survived the Zod migration**

This is the higher-value smoke test — it exercises Zod schema → JSON Schema conversion through the MCP SDK. If any of our migrated schemas (`rich-text`, `file`, `page-properties`, `icon`) break under Zod 4, `tools/list` is where it surfaces.

Run:

```bash
{
  echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"smoke","version":"0"}}}';
  sleep 0.5;
  echo '{"jsonrpc":"2.0","method":"notifications/initialized"}';
  sleep 0.5;
  echo '{"jsonrpc":"2.0","id":2,"method":"tools/list"}';
  sleep 0.5;
} | PATH="/Applications/OrbStack.app/Contents/MacOS/xbin:$PATH" \
  ~/.orbstack/bin/docker run --rm -i -e NOTION_TOKEN=ntn_fake_smoke notion-mcp-server:1.4.0 2>/dev/null \
  | python3 -c "
import sys, json
tool_count = None
for line in sys.stdin:
    if not line.strip(): continue
    msg = json.loads(line)
    if msg.get('id') == 2:
        tools = msg.get('result', {}).get('tools', [])
        tool_count = len(tools)
        print(f'tool count: {tool_count}')
        for t in tools:
            schema_ok = 'inputSchema' in t and isinstance(t['inputSchema'], dict)
            print(f'  - {t[\"name\"]}: schema_ok={schema_ok}')
        break
assert tool_count and tool_count >= 5, f'expected ≥5 tools, got {tool_count}'
print('OK')
"
```

Expected: 5 tools listed (`notion_pages`, `notion_blocks`, `notion_database`, `notion_comments`, `notion_users`), each with `schema_ok=True`, ending in `OK`.

If any tool reports `schema_ok=False`, the MCP SDK could not derive a JSON Schema from one of the Zod 4 schemas — investigate which one by inspecting the failing tool's tool-definition file under `src/tools/`.
