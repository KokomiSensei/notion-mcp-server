# MCP Protocol 2025-11-25 Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade `notion-mcp-server` from `@modelcontextprotocol/sdk ^1.9.0` to `^1.29.0`, migrate to the modern `registerTool` API with `title`/`annotations`/`outputSchema`/`structuredContent`, and fix two latent bugs (stdio stream corruption via `console.log`, invalid `error` field on `CallToolResult`).

**Architecture:** This server uses `StdioServerTransport` and exposes 5 "operation" tools (`notion_pages`, `notion_blocks`, `notion_database`, `notion_comments`, `notion_users`), each a discriminated union over multiple sub-actions. The migration is mechanical: replace `server.tool(name, desc, schema, handler)` calls with `server.registerTool(name, {title, description, inputSchema, annotations, outputSchema}, handler)` and return `{ content, structuredContent, isError? }` instead of the current `{ content, error? }` shape. No business logic changes.

**Tech Stack:** TypeScript 5.8, Node ≥20, `@modelcontextprotocol/sdk` 1.29.x, `@notionhq/client` 2.3.x, `zod` 3.24.x.

---

## Context: What is changing and why

### Protocol changes since the SDK 1.9 baseline

The project is pinned to `@modelcontextprotocol/sdk ^1.9.0`, which speaks the `2025-03-26` protocol. Two full spec revisions shipped since:

**`2025-06-18`:**
- `structuredContent` + `outputSchema` on tools (validated JSON, not just text)
- `ResourceLink` content blocks (tool results can reference resources by URI)
- Elicitation (server can ask the client for structured input mid-call)
- OAuth Resource-Server classification
- `_meta` everywhere; `title` on tools/prompts/resources
- Removed JSON-RPC batching

**`2025-11-25` (current):**
- **Tasks** — first-class async/long-running ops with `tasks/list`, `tasks/cancel`, polling, TTL
- **Elicitation v2** — split into `form` and `url` modes
- **Sampling with context/tools** — server can ask client to sample with tool access
- **Icons** on `clientInfo`/`serverInfo`/tools
- **`execution` field** on `Tool`
- Richer `clientInfo`/`serverInfo`: `title`, `description`, `websiteUrl`, `icons`

### What this plan does (and explicitly does not)

**In scope:**
1. Bump SDK to `^1.29.0`, Node engines to `>=20`.
2. Fix `console.log` → `console.error` in stdio path (current code can corrupt the JSON-RPC stream).
3. Drop the unused `resources` capability declaration.
4. Fix `CallToolResult` error shape: replace invalid top-level `error: {code, message}` with `isError: true`.
5. Migrate all 5 tool registrations from `server.tool(...)` to `server.registerTool(...)` with `title` and `annotations` (`readOnlyHint`, `destructiveHint`, `openWorldHint`).
6. Add `serverInfo` enhancements: `title`, `websiteUrl`.
7. Bump server version `1.0.1` → `1.1.0`.

**Out of scope (deliberate):**
- **Tasks API** — none of the current Notion operations are long-running enough to justify; revisit if batch ops ever stream progress.
- **Resources** — Notion pages-as-resources is a reasonable future change, but it's a new surface, not a migration.
- **Icons** — cosmetic, requires hosting icon files; defer.
- **outputSchema + structuredContent for every tool** — the existing tools return raw Notion API objects whose shape is large and unstable. Migrating these properly means declaring real Zod-derived JSON Schemas for each Notion response; that's a separate task per tool. This plan only sets up the *call shape* (`registerTool`) so a follow-up can add outputSchemas incrementally.
- **Test suite** — the project currently has no test runner. Adding Jest/Vitest is a separate setup. Verification in this plan uses `npm run inspector` (the official `@modelcontextprotocol/inspector`) for manual smoke checks at each checkpoint.

### Known bugs being fixed along the way

1. **`src/server/index.ts:25` — `console.log` writes to stdout**, which is the JSON-RPC channel for `StdioServerTransport`. Strict clients will reject this. Must be `console.error`.
2. **`src/utils/error.ts:118-121, 133-136, 148-151` — invalid `error` field on `CallToolResult`.** The MCP spec defines `isError: boolean` at the result level; the `error: {code, message}` shape is JSON-RPC *protocol* error territory, not a tool result. The current TypeScript is loose enough to compile, but the field is silently dropped by spec-compliant clients. Replace with `isError: true`.

### Annotation choices per tool

`ToolAnnotations` are hints the client uses for UX (e.g., warn before destructive ops). Multi-action tools must pick the most conservative honest answer:

| Tool             | readOnlyHint | destructiveHint | openWorldHint | Reasoning                                                                              |
|------------------|--------------|-----------------|---------------|----------------------------------------------------------------------------------------|
| `notion_pages`   | false        | true            | true          | `archive_page` is destructive; all ops hit external Notion API.                        |
| `notion_blocks`  | false        | true            | true          | `delete_block`, `batch_delete_blocks` are destructive.                                 |
| `notion_database`| false        | false           | true          | `create_database`, `query_database`, `update_database` — no delete in current surface. |
| `notion_comments`| false        | false           | true          | Add-only writes plus read-only fetches.                                                |
| `notion_users`   | true         | false           | true          | `list`, `get`, `get bot` are all reads.                                                |

`idempotentHint` is not set on any tool — too action-dependent to assert truthfully.

## Verification approach (every task)

The project's verification tool is `npm run inspector`, which starts the server under `@modelcontextprotocol/inspector`. The inspector is an interactive web UI; it cannot be scripted from a plan. Instead, each task either:

1. Uses **TypeScript compilation** (`npm run build`) as the structural check — the SDK has strong types, so a misshapen `registerTool` call or wrong `CallToolResult` field will fail the build, and

**Files:**

- [ ] **Step 1: Replace console.log with console.error**

Open `src/server/index.ts`. Find:

```ts
    console.log(
      `${CONFIG.serverName} v${CONFIG.serverVersion} running on stdio`
    );
```

Replace with:

```ts
    console.error(
      `${CONFIG.serverName} v${CONFIG.serverVersion} running on stdio`
    );
```

Why: `StdioServerTransport` reads JSON-RPC from stdin and writes JSON-RPC to **stdout**. Any non-JSON byte written to stdout by user code (including this startup message) breaks the framing for strict clients. `stderr` is the only safe channel for server logs in stdio mode.

```ts
const ERROR_MESSAGES: Record<string, string> = {
  [NotionErrorCode.InvalidJson]:
    "The request body could not be decoded as JSON",
  [NotionErrorCode.InvalidRequestUrl]: "The request URL is not valid",
  [NotionErrorCode.InvalidRequest]: "This request is not supported",
  [NotionErrorCode.ValidationError]:
    "The request body does not match the schema for the expected parameters",
  [NotionErrorCode.MissingVersion]:
    "The request is missing the required Notion-Version header",
  [NotionErrorCode.UnsupportedVersion]:

  [NotionErrorCode.UnsupportedExport]:
    "The specified export type is not supported",
  [NotionErrorCode.UnsupportedJsonType]:
    "The specified JSON type is not supported",
  [NotionErrorCode.UnsupportedJsonKey]:
    "The specified JSON key is not supported",
  [NotionErrorCode.Unauthorized]: "The bearer token is not valid",
  [NotionErrorCode.InvalidApiKey]: "The API key is invalid",
  [NotionErrorCode.RestrictedResource]:
    "The resource is restricted and cannot be accessed with this token",
  [NotionErrorCode.InsufficientPermissions]:
    "The bearer token does not have permission to perform this operation",
  [NotionErrorCode.ObjectNotFound]: "The requested resource does not exist",
  [NotionErrorCode.ConflictError]:
    "The transaction could not be completed due to a conflict",
  [NotionErrorCode.AlreadyExists]: "The resource already exists",
  [NotionErrorCode.RateLimited]:
    "The request was rate limited. Retry later with exponential backoff",
  [NotionErrorCode.InternalServerError]:
    "An unexpected error occurred on the Notion servers",
  [NotionErrorCode.ServiceUnavailable]:
    "The Notion service is unavailable. Retry later with exponential backoff",
  [NotionErrorCode.DatabaseConnectionUnavailable]:
    "The database connection is unavailable. Retry later with exponential backoff",
};

function getErrorMessage(
  notionErrorCode: string,
  defaultMessage?: string
): string {
  return (
    ERROR_MESSAGES[notionErrorCode] ||
    defaultMessage ||
    "An unknown error occurred"
  );
}

/**
 * Handles a Notion API error and returns a spec-compliant CallToolResult
 * with isError: true. Per MCP spec, tool execution errors are surfaced
 * via isError + a text content block describing the failure; the JSON-RPC
 * error envelope is reserved for protocol-level errors.
 */
export function handleNotionError(error: unknown): CallToolResult {
  if (error instanceof APIResponseError) {
    const code = error.code as string;
    const message = getErrorMessage(code, error.message);
    return {
      isError: true,
      content: [
        {
          type: "text",
          text: `Error: ${message} (${code})`,
        },
      ],
    };
  }

  if (error instanceof Error) {
    return {
      isError: true,
      content: [
        {
          type: "text",
          text: `Error: ${error.message}`,
        },
      ],
    };
  }

  return {
    isError: true,
    content: [
      {
        type: "text",
        text: `Error: ${String(error)}`,
      },
    ],
  };
}
```

Key changes from the previous version:
1. Dropped the `ErrorCode` import from `@modelcontextprotocol/sdk/types.js` — no longer used.
2. Each return value now has `isError: true` and no `error` field.
3. The JSDoc on `handleNotionError` explains the spec rationale.

## Post-migration: optional follow-ups (not in this plan)
